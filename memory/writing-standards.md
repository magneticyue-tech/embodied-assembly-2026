# 写作规范

适用范围:本仓库所有文档、代码注释、提交信息、答辩材料,以及助手在对话中对用户的输出。

## 规则

1. **禁止绝对化表述。** 不使用"无缝、完美、彻底、100%、真正的、本质上、绝不"等词修饰自己的工作。接口可替换 ≠ "无缝互换";写"更换实现时主流程不变,行为一致性需联调验证"。
2. **事实描述与价值判断分离。** 实现描述里不夹带评价。反例:"真正的精度瓶颈在视觉"(判断)混在硬件参数注释(事实)里;应只写参数,若需给出判断,单独成段并标明是推断。
3. **性能结论必须有量化数据支撑。** 写明测量条件:样本量、随机种子、环境版本。无数据的结论标注"未验证"。反例:"定位误差<0.5mm"未记录测量条件,不得作为已确认事实引用。
4. **事实先行,推论后置。** 先陈述文档/代码/数据中实际存在的内容,再给推断或建议,二者分开陈述。
5. **区分观测与推断。** "文档里有什么"与"由此推断什么"不混为一谈;缺失的内容单独列出("暂无 X"),不强行归类。
6. **标题中不夹带价值判断。** 用中性标题("仿真边界"),不用编辑化标题("仿真边界(答辩要诚实说明)")。
7. **不为迎合读者偏好添加括号附注或结论。** 不写"(这是你的得分点)"一类话。
8. **汇报工作不自我标榜。** 直接陈述验证了什么、结果如何、还有什么未覆盖或存疑;不堆砌验证手段数量,不用表功式措辞。写"跑通了,退出码 0;相机噪声致环数 ±1 抖动未消除",不写"通过全部验证"。
9. **搬运旧文档措辞时,先按本规范重写再落笔。**
10. **本项目文档用中文。** 代码标识符与日志键名用英文。
11. **不写博取注意、对信息传递无增益的开场。** 禁用"核心事实一句话""一句话说清""关键在于"等引子,直接陈述内容。反例:"核心事实一句话:正方形和长方形对称性不同",应写"正方形和长方形对称性不同"。
12. **优先正向表达,不用先否定再肯定的戏剧化结构。** 反例:"这不是 X,而是 Y";应直接写"这是 Y"。
13. **克制关联词。** 不滥用"因此、所以、然而、不仅…而且"堆叠因果与转折;能拆成短句陈述的,分句直述。
14. **不加引导注意的附注或排序性修饰。** 禁用"(清单里最挡路的)""(重点)""(关键)"一类括注去替读者排优先级;要突出某项,直接陈述其技术后果,由读者判断轻重。
15. **不猜测未提供的内容。** 文档/代码/对话里没有的,不臆测填充;残缺、截断、未定义处如实标为"待补/待确认",不自造。
16. **用中性动词替代"定死"等收口词。** 表示"确定下来"用"确认、审核、核查、定稿";不用"定死"。

**Why:** 校赛为 PPT 答辩,语言质量被评审;绝对化与表功措辞不增加可信度,且掩盖真实边界,误导后续工程决策。

**How to apply:** 写作前后各过一遍上述清单;评审时(见 [[collaboration-protocol]])将措辞违例与技术错误同等对待。
