# 具身智能精密装配赛 — 正向设计方案

> 本文档用于团队内部理解实现细节,采用正向设计思路:需求 → 技术指标 → 总体架构 → 模块详细设计 → 关键算法 → 验证方案。
> 适用赛题:2026 中国大学生机械工程创新创意大赛 · 具身智能精密装配赛。
> 注:文中标注「待确认」处需结合培训视频、赛方器材清单或现场调试确定。

**已确认工具链(来自赛方资料):**
- 机械臂:**AUBO-i5**(6 自由度,负载 5kg,工作半径 886.5mm,重复定位精度 **±0.02mm**;SDK 支持 Python/C/C++/Java/Lua)。
- 机器人仿真:**ARCS / AUBOPE**(官方,VMware 虚拟机部署,支持 Python 脚本、图形化编程、轨迹记录回放)。
- 托盘:**同心环精度尺**(每环 1mm、每 5mm 加粗、量程 ±20mm),放置偏移按落在第几环读数评分。
- 可选视觉平台:海康 **Vision Master**(无代码拖拽);本方案选**自写 OpenCV**(理由见 §3.0)。
- 任务卡指令来源:**相机视觉读卡**(VLM 解析),非裁判口述;语音仅负责唤醒/触发/复述。

---

## 1. 需求分析与技术指标推导

从任务书反推出可落地的技术指标。正向设计的第一步是把"赛题要求"翻译成"工程约束"。

| 赛题原始要求 | 推导出的技术指标 / 约束 |
|---|---|
| 方块边长 30mm,放入托盘对应颜色框,放置后稳定不可再调 | 放置高度控制避免冲击;AUBO-i5 自身 ±0.02mm 远超需求 → **精度瓶颈在视觉定位** |
| 托盘为同心环精度尺(1mm/环,±20mm) | 放置偏移可被裁判读数 → 视觉定位需做到**环级精度**;视觉误差直接对应得分 |
| 单目相机视觉检测 | 无原生深度,需用平面约束+已知尺寸恢复 3D;依赖标定精度 |
| 方块/托盘摆放可能旋转(边不与画面平行) | 检测需旋转鲁棒:用 minAreaRect 取中心+角度;吸盘放置需角度对齐 |
| 6 色方块(红橙黄绿蓝紫)识别 | 颜色空间需区分相近色(红/橙、蓝/紫);受控光源下做分割 |
| 任务卡随机抽取、内容随机 | 不能硬编码;识别需泛化;需置信度校验与失败处理 |
| 机器人前往拍照点期间裁判可再次挪动方块/托盘 | 必须「每块装配前重新感知」的闭环,不可一次扫描定全场 |
| 现场不提供网络 | 大模型与语音需离线部署,或自备网络(性能/稳定性权衡) |
| 唤醒词「小具同学」、应答「我已就绪,请下达指令」 | 唤醒词可定制的离线 KWS 引擎;固定话术 TTS |
| 解析过程保存带时间戳文本 | 全链路日志系统,每步落盘,含时间戳 |
| 识别失败语音提示「任务卡识别失败,请确认任务卡位置」 | 置信度阈值 + 失败分支 + 对应 TTS |
| 全自动、宣告就绪后不可干预、5 分钟、失败重抽取最优 | 高鲁棒状态机;异常自恢复;无人工兜底 |
| 任务一、二无先后,都要做,各自完成播报「任务已完成」,全部完成播报「两个任务均已完成」 | 双任务调度;完成事件触发对应播报 |

**核心设计原则(贯穿全方案):**
> 语义任务交给大模型,度量任务交给经典 CV + 几何。
> 大模型负责"理解任务卡说了什么、规划先放哪块",不输出抓取坐标;精确定位由颜色分割 + 平面反投影计算得出。这是区分可行方案与不可行方案的关键。

## 2. 系统总体架构

三层架构,自上而下:

```
┌─────────────────────────────────────────────┐
│  认知层 (大脑)  大模型智能体 Agent              │
│  · 任务卡识别/场景理解  · 指令解析→结构化JSON    │
│  · 任务规划(装配顺序)   · 对话与播报话术生成     │
└───────────────┬─────────────────────────────┘
        指令/规划 ↓        ↑ 感知结果/状态
┌───────────────┴─────────────────────────────┐
│  感知层 (感官)                                │
│  · 视觉: 标定→拍照→颜色分割→定位→反投影         │
│  · 语音: 唤醒(KWS)→识别(ASR)→合成(TTS)        │
└───────────────┬─────────────────────────────┘
        目标世界坐标 ↓     ↑ 执行反馈
┌───────────────┴─────────────────────────────┐
│  执行层 (小脑)  机器人控制                      │
│  · 坐标变换  · 运动规划  · 吸盘抓放  · 顺序控制  │
└─────────────────────────────────────────────┘
```

**数据流(以任务二一次装配为例):**
1. 语音唤醒 → ASR 触发 → Agent 调用视觉读任务卡 → 输出结构化装配序列 JSON
2. Agent 语音复述指令 → 取序列第 1 步(如"红块→黄盘")
3. 执行层移至拍照点 → 视觉对当前场景定位红块与黄盘 → 反投影出世界坐标
4. 运动规划 → 吸盘抓红块 → 放至黄盘 → 反馈成功
5. 回到第 3 步处理下一块(每块重新感知,兼容中途挪动)
6. 6 块完成 → 播报"任务已完成"

**模块间通信:** 视觉(Python/OpenCV)、Agent、机器人控制走本地进程通信(socket/gRPC 或共享消息);机器人侧用 **AUBO SDK**(Python)下发运动指令。大模型本地部署,Agent 侧本地调用。全本地化,不依赖现场网络。

## 3. 感知层 · 视觉模块详细设计

视觉是整套方案精度的根基。分五个子环节:标定 → 拍照 → 颜色识别 → 定位 → 反投影。

### 3.0 视觉平台选型:自写 OpenCV vs Vision Master

赛方提供海康 **Vision Master**(无代码拖拽式视觉平台)。本方案选**自写 OpenCV**,理由:
- **旋转鲁棒**:方块/托盘可能旋转,需 minAreaRect 取中心+角度并按 90° 对称折叠,自写更可控。
- **同心环误差读数**:把放置偏移换算成"第几环",与裁判评分口径对齐,需自定义逻辑。
- **与大模型紧耦合**:视觉结果直接喂 Agent,流程统一。
- **全管线可控可解释**:满足评分细则"实时展示坐标提取过程"的可观测要求。
> 取舍:Vision Master 开发快、易上手,适合快速搭建标准定位;若团队 OpenCV 经验有限,可作为备选或交叉验证。

### 3.1 相机标定(离线一次性,赛前完成)

**相机内参标定**:用棋盘格标定板,OpenCV `calibrateCamera` 求出内参矩阵 K 与畸变系数。
- 内参矩阵 K = [[fx, 0, cx], [0, fy, cy], [0, 0, 1]],fx/fy 为焦距(像素),cx/cy 为主点。
- 目的:把畸变后的像素坐标矫正,并支持像素↔归一化平面的转换。

**手眼标定(关键)**:相机挂载在机械臂末端(eye-in-hand),需求出相机坐标系与机器人末端坐标系之间的固定变换 X(齐次变换矩阵 4×4)。
- 经典方法:控制机械臂以多个不同姿态拍同一标定板,采集多组 (机器人末端位姿 A_i, 标定板相对相机位姿 B_i),求解 AX = XB(OpenCV `calibrateHandEye`)。
- 输出 X 后,任意时刻已知机器人末端位姿,即可把相机坐标系下的点变换到机器人基坐标系。
- 精度直接决定抓取成败,赛前需反复验证,现场也保留快速校准流程。

### 3.2 拍照策略

- 设置一个或多个固定「拍照点」(机器人位姿),保证台面物料在视野内、光照均匀。
- 每次装配前回到拍照点重新拍照(闭环再感知,应对裁判挪动)。
- 配合光源消除阴影与反光;曝光/白平衡固定,避免颜色漂移。

### 3.3 颜色识别(6 色分割)

**颜色空间选择**:不要用 RGB(对光照敏感)。用 **HSV** 或 **Lab**:
- HSV:H(色相)区分颜色、S(饱和度)、V(明度)。色相对光照鲁棒。
- 相近色处理:红(H≈0/180)、橙(H≈10~25)、黄(H≈25~35)、绿(H≈35~85)、蓝(H≈85~125)、紫(H≈125~155)需在受控光源下实测标定各色 H 范围,留出间隔避免误判。红色跨 0/180 需用两段区间。
- 建议赛前在实训台上对 6 色方块逐一采样,统计各色 HSV 分布,固化阈值表。

**分割流程**:
1. 转 HSV → 对每个颜色用 `inRange` 得到二值掩膜
2. 形态学开运算去噪 → 闭运算补洞
3. `findContours` 找轮廓 → 按面积过滤(滤掉噪点和非方块)
4. 对每个有效轮廓求**最小外接旋转矩形 `minAreaRect`** → 得中心 + 旋转角(图像坐标)

**旋转处理**:方块/托盘的边不一定与画面平行。用 minAreaRect 而非水平外接框,直接拿到中心与朝向。正方形 90° 旋转对称,角度折叠到 [-45°,45°] 再使用/比较。

**方块 vs 托盘区分**:托盘中心也是实心彩色方块(被同心环包围),与方块中心同色同形,故不能只靠填充率。判别法:在彩色方块**外围环带**统计边缘密度(Canny)——托盘外有同心灰环,边缘密度高;方块在空台面上,边缘密度低。30mm 方块在已知视距下的像素尺寸是已知量,可作面积强约束。(sim/vision.py 已验证此法)

### 3.4 像素 → 世界坐标(平面反投影,核心)

单目无深度,但所有方块和托盘都在**同一已知平面**(台面)上,这是破解深度的钥匙。

**方法 A:单应矩阵 H(推荐,简单稳健)**
- 台面是一个平面,图像平面到台面平面之间存在单应变换 H(3×3)。
- 赛前在台面上放标定板或已知坐标的标记点,采集 ≥4 组「像素坐标 ↔ 台面物理坐标」,`findHomography` 求 H。
- 运行时:任意方块/托盘质心像素 (u,v) → 经 H 映射 → 台面物理坐标 (X,Y)。Z 为已知台面高度(+ 半个方块高度作为抓取点)。
- 优点:不需要逐帧解 PnP,标定一次后映射极快且稳定。

**方法 B:已知尺寸 + PnP**
- 利用 30mm 方块的已知尺寸,对方块顶面四角解 PnP 求位姿。更通用但对角点检测精度敏感。
- 作为方法 A 的补充或交叉校验。

**得到台面坐标后** → 经手眼变换 X 与当前末端位姿,转为机器人基坐标系下的抓取目标。

**同心环误差读数(精度自检)**:托盘为同心环精度尺(1mm/环)。把"放置点相对托盘中心的偏移"换算成环数 `ring = round(offset_mm / 1mm)`,即可与裁判读数口径一致,实现放置精度的自评估与闭环。

### 3.5 任务卡识别

任务卡内容随机,不能硬编码。**装配指令由相机读卡、大模型解析得出**(非裁判口述)。两条路线:
- **路线 1(推荐)**:直接交给多模态大模型(VLM)。把任务卡照片喂给 VLM,让其输出结构化结果(场景描述 / 装配序列 JSON)。泛化性最好,契合"大模型智能体"考点。
- **路线 2**:传统 OCR(如 PaddleOCR)读文字 + 规则解析。离线、快,但对版式和泛化要求高。
- **失败校验**:无论哪条路线,需对结果做置信度判断(VLM 可要求其自评置信度/是否能清晰识别),低于阈值触发"任务卡识别失败,请确认任务卡位置"语音。

## 4. 感知层 · 语音模块详细设计

语音链路:唤醒(KWS)→ 识别(ASR)→ 合成(TTS),全程离线优先(现场无网)。

### 4.1 语音唤醒 KWS

- 唤醒词固定为「小具同学」,需可定制唤醒词的离线引擎。
- 候选:Picovoice Porcupine(可训自定义词,轻量)、Snowboy(已停维护但仍可用)、或国内方案如科大讯飞离线唤醒 SDK、ESPHome/snowman 等。
- 唤醒后立即用 TTS 播报固定话术「我已就绪,请下达指令」。

### 4.2 语音识别 ASR

- 离线候选:**FunASR**(阿里达摩院,中文效果好,可离线)、**Whisper / faster-whisper**(多语种,中文可用,GPU 加速)、Vosk(轻量离线)。
- **ASR 权重较低**:已确认装配指令来自**相机读卡**(VLM 解析),不是裁判口述。语音只负责唤醒、触发任务、自然语言交互;装配指令的解析不依赖 ASR。
- 输出文本送 Agent 解析。

### 4.3 语音合成 TTS

- 离线候选:edge-tts(需网,不适合)、**Piper**(离线、轻量、音质好)、PaddleSpeech TTS、或厂商离线 TTS SDK。
- 需播报的固定话术:应答语、指令复述、"任务已完成"、"两个任务均已完成"、识别失败提示等。
- 建议:固定话术可预生成音频文件直接播放(最稳),动态内容(如复述识别到的装配序列)才实时合成。

### 4.4 备用文本输入

- 规则要求必须具备文本输入功能,仅作语音失败时备用,使用前需向裁判申请,每条扣分。
- 实现为一个简单输入框,接同一套解析逻辑。**禁止**按键/触摸屏/串口等非指定方式——文本输入的具体合规形式需对照规则与培训确认。「待确认」

## 5. 认知层 · 大模型智能体详细设计

### 5.1 模型选型(本地 vs 云端)

| 方案 | 优点 | 缺点 | 适用 |
|---|---|---|---|
| 本地多模态 VLM(如 Qwen-VL / Qwen2.5-VL 量化版) | 离线、无网络依赖、稳定 | 需较强 GPU/边缘算力;部署复杂 | 现场无网,推荐 |
| 云端 API(需自备网络) | 能力强、部署省事 | 依赖自备网络稳定性;有延迟 | 自备稳定 4G/5G 时可选 |

任务书允许本地开源模型或云端 API,网络自备。**本方案采用 Claude 大模型**:一个多模态模型同时承担任务卡识别、场景理解、指令解析三件事,减少环节。语音唤醒/合成在本地运行,现场网络自备以保证大模型稳定调用。AI 边缘计算平台的算力上限需确认。「待确认」

### 5.2 智能体的职责边界(重申核心原则)

Agent **做**:理解任务卡语义、把自然语言/卡面指令解析为结构化 JSON、规划装配顺序、生成对话与播报话术、对感知结果做合理性校验。
Agent **不做**:不输出像素/世界坐标,不做几何计算——这些交给视觉模块。

### 5.3 结构化输出契约

让大模型输出严格 JSON,执行层只认 JSON,避免自然语言歧义。示例(任务二):

```json
{
  "task": "assembly",
  "confidence": 0.97,
  "sequence": [
    {"step": 1, "block_color": "red",    "tray_color": "yellow"},
    {"step": 2, "block_color": "green",  "tray_color": "red"},
    {"step": 3, "block_color": "orange", "tray_color": "blue"},
    {"step": 4, "block_color": "blue",   "tray_color": "purple"},
    {"step": 5, "block_color": "yellow", "tray_color": "orange"},
    {"step": 6, "block_color": "purple", "tray_color": "green"}
  ]
}
```

- 用 prompt 约束输出格式;可加 JSON Schema 校验,解析失败则重试或走失败分支。
- `confidence` 低于阈值 → 触发识别失败语音。

### 5.4 提示词工程要点

- 明确角色与任务、给出颜色枚举(红橙黄绿蓝紫)、强制 JSON 输出、给 few-shot 示例。
- 对场景识别(任务一)和指令解析(任务二)用不同 prompt 模板。
- 要求模型对模糊/遮挡情况主动报告而非编造。

### 5.5 日志(强制要求)

解析过程必须以文本保存且带时间戳。每次大模型调用记录:时间戳、输入(图片路径/文本)、输出 JSON、置信度。落盘为日志文件,答辩时也可作为可解释性展示。

## 6. 执行层 · 机器人控制详细设计

### 6.1 坐标系与变换链

完整变换链(把视觉看到的方块变成机械臂能去抓的位姿):

```
像素坐标 (u,v)
   │  内参 K + 畸变矫正
   ▼
归一化图像坐标
   │  单应 H(平面反投影,3.4 节)
   ▼
台面物理坐标 (X,Y),Z=台面高度
   │  相机坐标系下表示
   ▼
   │  手眼变换 X(3.1 节)+ 当前末端位姿 T_base→ee
   ▼
机器人基坐标系下抓取目标位姿
```

关键:每个变换矩阵都要在赛前标定好并验证。任何一环误差都会累积到末端。

### 6.2 抓取与放置(吸盘)

- 末端是电动吸盘,抓取正方体顶面中心。抓取点 = 方块质心的 (X,Y) + 顶面高度 Z。
- **角度对齐**:用视觉给出的旋转角对齐末端,使方块边与托盘色框对齐(放置更稳、更准入框)。
- **抓取流程**:移动到目标上方安全高度 → 垂直下降到接触高度 → 开吸盘 → 上抬 → 移动到托盘上方 → 下降到放置高度 → 关吸盘释放 → 上抬。
- **放置约束**:方块须落入托盘对应颜色框、放后稳定不倾倒、不可二次触碰。所以放置高度和释放时机要精确,避免砸落或偏移。
- 30mm 方块、吸盘需对准中心,定位误差预算要留够;AUBO-i5 自身 ±0.02mm,误差主要来自视觉。

### 6.3 运动规划

- 简单点位运动用 **AUBO SDK** 的关节/直线运动指令(MoveJ/MoveL)。
- 需设置安全高度过渡点,避免碰撞已放置的方块和托盘。
- 路径中加入避让逻辑:已装配区域的方块不可碰(规则硬约束)。
- 机器人为 **AUBO-i5**,SDK 支持 Python/C/C++/Java/Lua;赛前用官方 **ARCS/AUBOPE** 仿真(VMware + Python 脚本 + 轨迹记录回放)跑通运动。

### 6.4 双任务调度与状态机

任务一、二无先后,都要做;每次完成播报,全部完成总播报。用状态机管理:

```
[空闲] --唤醒"小具同学"--> [就绪:播报应答]
   │
   ├─任务卡1--> [场景识别] --成功--> [播报"任务已完成"] ─┐
   │                       └失败--> [重抽/重启程序]      │
   │                                                     ▼
   ├─任务卡2--> [指令解析] --> [循环装配6块] --成功--> [播报"任务已完成"]
   │              每块: 拍照→定位→抓→放(闭环再感知)        │
   │                       └任一步失败--> [记分后重启]      │
   │                                                     ▼
   └──────────────> [两任务均完成?] --是--> [播报"两个任务均已完成"]
```

- 单次任务失败:按规则记分后执行另一个任务,不可调试,重启程序。
- 5 分钟内可重抽任务卡重来,取最优成绩 → 程序需支持快速复位到初始状态。

| 系统文件误删致机器人崩溃(0分) | 严格备份;不动系统文件;按培训视频规范操作 |

## 8. 验证方案(无需真机也能推进)

正向设计强调"每个模块独立可验证"。在拿到真机前的验证路径:

| 模块 | 验证方式 | 是否需真机 |
|---|---|---|
| 任务卡识别 / 指令解析 | 拍真实任务卡照片(或自制样卡)喂 VLM,核对 JSON | 否 |
| 颜色分割 / 定位 | 拍彩色方块+托盘照片,跑 OpenCV 管线看结果 | 否 |
| 平面反投影精度 | 用标定板照片,核对反投影坐标与真值误差 | 否(需照片) |
| 语音唤醒/ASR/TTS | 麦克风+音箱本地测,念唤醒词、测识别率 | 否 |
| 运动/抓放流程 | **ARCS/AUBOPE** 官方仿真跑通逻辑 | 否(仿真) |
| 视觉+认知+调度逻辑 | 本项目 **sim/**(纯软件端到端) | 否 |
| 全链路联调 | ARCS 仿真或真机端到端 | 真机最终确认 |

**仿真平台**:
- 运动/抓放:官方 **ARCS / AUBOPE**(VMware 虚拟机、Python 脚本、轨迹记录回放),最贴近 AUBO-i5 真机。
- 视觉/认知/调度逻辑:本项目 `sim/` 已实现纯软件端到端复现(旋转鲁棒检测 + 同心环误差读数 + 时间戳日志),无需真机/GPU。
- 认知层与视觉也可用真实照片离线测;追求高保真渲染可选 NVIDIA Isaac Sim(需较强 GPU,非必需)。
